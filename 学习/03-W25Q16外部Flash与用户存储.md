## 专有名词

| 名词 | 解释 |
|------|------|
| **SPI** | 4 线串行：SCK 时钟、MOSI 主出、MISO 主入、CS 片选 |
| **软 SPI** | 用 GPIO 位操作模拟 SPI，不占硬件 SPI 外设 |
| **JEDEC ID** | 上电读的芯片型号 ID，W25Q16 常见 `EF4015` |
| **扇区擦除** | Flash 只能 1→0；擦除后变 0xFF 才能再写 |
| **页写** | W25Q 一次最多写 256 字节 |
| **魔数 Magic** | 数据区头部标记，如用户表 `"USR1"`，防止读错区域 |

---

## 功能怎么实现

### 1. 驱动层

`bsp_w25q16.c` 用软 SPI 实现：

- 读 JEDEC ID（确认芯片在线）
- 扇区擦除（4KB）
- 页编程、连续读

主工程引脚：**CS=PA15, SCK=PB3, MISO=PB4, MOSI=PB5**

### 2. 存储布局（`app_user_ops.c`）

| 区域 | 地址 | 魔数 | 用途 |
|------|------|------|------|
| 用户表 | `0x00000000` | `USR1` | 最多 64 个 `user_cred_t` |
| 指纹模板缓存 | `0x00180000` | `FPT1` | 主机镜像用 |

### 3. 启动顺序

`main.c` 在以下操作**之前**调用 `users_storage_load()`：

- LVGL 初始化（LCD 占 SPI1）
- 长时间 UI 操作

原因：避免总线冲突；PA15 默认 JTAG，需尽早读完释放相关引脚状态。

### 4. 异步刷盘

用户增删改后标记 dirty，由 **StorageTask**（低优先级）异步写入，避免在 GUI 线程里擦除卡死界面。

## 动手实验

1. 用第 2 课方法跑 W25Q 自检，确认 `EF4015`。
2. 在主工程打开 `APP_W25Q_DEBUG=1`（若 Flash 允许），观察加载日志。
3. 在 `app_user_ops.c` 搜索 `USR1`，画出用户表头部结构。

---

## 易错点 / 困难点

- **PA15 是 JTAG 引脚**，调试需 SW 模式，否则读不到 Flash。
- **擦除慢**（几十 ms），禁止在 GuiTask 里同步大块擦写。
- 读失败或没有可登录管理员时，代码会 RAM 补默认 `1/1111`，**不会清空已有数据**。
- 测试板 PE0 与主工程 PA15 不同，换板必查原理图。

---

## 本课总结

- W25Q16 = 掉电保存的用户表与指纹模板缓存
- 软 SPI + 扇区擦除 + 页写
- 启动**尽早** load 用户表
- 写 Flash 走 StorageTask 异步路径

---

## 本课提问

1. 用户表在 Flash 的起始地址和魔数是什么？
2. 为什么 main 里要尽早 load 用户？
3. 主工程 W25Q 四根 SPI 线接哪几个脚？
4. 改密码后何时真正写入 W25Q？

---

## 参考答案

1. 起始地址 `0x00000000`，魔数 4 字节 `"USR1"`。
2. 避免与 JTAG/SPI1(LCD) 争用；尽早有用户数据供鉴权；保证后续 NFC/指纹等初始化顺序合理。
3. CS=PA15，SCK=PB3，MISO=PB4，MOSI=PB5。
4. 修改后标记 dirty，由 Storage 任务异步刷盘，不在 UI 线程同步擦写。
