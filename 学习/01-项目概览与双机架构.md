## 通俗讲解

把这套系统想成**一套双门智能锁**：

| 角色 | 比喻 | 实际能力 |
|------|------|----------|
| **Host（主机）** | 大门 + 管家 | 有屏幕、能连 WiFi、管所有用户、录入 NFC/指纹、同步给从机、上报云端 |
| **Client（从机）** | 侧门保安 | 只有简单开锁界面，本地 NFC/指纹/密码开锁，**没有 WiFi**，开锁结果通过 RS485 告诉主机上云 |

两块板子**硬件相同**，靠编译时改一个宏变成主机或从机——这叫「**一板多镜像**」，产品里很常见。

---

## 专有名词（本课细讲）

| 名词 | 通俗解释 | 在本项目里 |
|------|----------|------------|
| **固件** | 烧进 MCU 里的程序 | Host/Client 各一份 Keil 工程 |
| **BSP** | 板级支持包，驱动具体硬件 | `Host/Drivers/BSP/` |
| **HAL** | ST 官方硬件抽象层 | `STM32F4xx_HAL_Driver/` |
| **Keil MDK** | 编译 + 烧录工具 | `*.uvprojx` 工程文件 |
| **RS485** | 两根线传数据，可拉长距离、抗干扰 | 主机地址 `0x01`，从机 `0x02` |
| **物模型** | 阿里云定义的设备属性字段 | `unlock_method`、`unlock_time` 等 |
| **MQTT** | 轻量消息协议，门锁经 ESP8266 连阿里云 | `cloud_aliyun_at.c` |

## 功能怎么实现的（项目视角）

```
用户在 Host 录入账号/NFC/指纹
    → 写入 W25Q Flash
    → RS485 镜像到 Client（用户 + 指纹模板分片）
    → Client 侧门刷卡/指纹 → 本地验证 → RS485 通知 Host
    → Host MQTT 上报 → Node 后端 → JoyfulZone App
```

手机远程开锁、临时密码、配对绑定走另一条路：

```
App → 后端 server/get → 云流转 → 门锁 terminal/get → Host 执行
```

## 易错点 / 困难点

- **烧错固件**：两块板都烧 Host，RS485 对不上，从机无响应。
- **以为 Client 也能配 WiFi**——不能，云功能只在 Host。
- **`app_config.h` 里阿里云密钥**是示例，真部署要换自己的，且勿提交 Git。

---

## 本课总结

- 项目是 **STM32F407 双机智能门锁**，配套 JoyfulZone App。
- **Host** = 全功能主控；**Client** = 侧门从机。
- 同一硬件，**宏切换角色**（`APP_RS485_NODE_ROLE`）。
- 数据流：录入 → Flash → RS485 同步 → 开锁 → 云端 → App。

4. 为什么 README 说有两套测试工程路径？

---

## 参考答案

1. Host 有完整 UI、WiFi/云、用户管理、向从机同步；Client 只有侧门开锁和 RS485，无云。
2. 变成 RS485 从机（地址 `0x02`），应烧 `Client/Projects/MDK-ARM/` 工程，或在 Host 工程改宏为从机后重新编译。
3. Client 发 `SLAVE_UNLOCK_NOTIFY`（RS485）→ Host 接收 → MQTT `terminal/push` → 后端 → App。
4. `test/test/` 是独立推荐的一体自检；`Host/test/nfc_enroll_rs485/` 是 Host 目录下的 RS485 专项测试，功能类似但路径不同。
