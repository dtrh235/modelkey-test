## 完整用户旅程

```
1. 管理员 WiFi 配网（UI3_SCR_WIFI）
      → MQTT 在线

2. App 生成配对码 → 门锁 UI3_SCR_PAIR 输入
      → app_pair_bind 完成 bind
      → 后端记录设备与 App 关系

3. 添加用户 / 录 NFC / 录指纹
      → Flash + RS485 镜像 Client

4. 下发临时密码（App 或管理员）
      → Host 存储 + RS485 TEMP_PWD_SET → Client

5. 任一门开锁
      → UART4 开门 + 记录上云 → App 可见
```

---

## 配对（Pair）

- 门锁屏显示/输入约 6 位配对码
- 经 `terminal/push` 或物模型与后端同步
- App 确认后建立「这台门锁 ↔ 这个用户/家庭」关系
- 具体字段见 `app_pair_bind.c` 与 JoyfulZone 后端 API

---

## 临时密码

### 主机侧

- `app_temp_password.c`：限时账号密码、过期撤销
- 开锁时 `method_id = 5`
- RS485 同步从机：`TEMP_PWD_SET`（account+password+valid_sec 小端）、`TEMP_PWD_REVOKE`

### 注意

- `valid_sec` 为 **uint32 小端**
- 过期后须撤销，否则从机仍可能校验旧临时口令（取决于同步状态）
- 上报账号可能是 temporary account 字段，见 proto 注释

## 动手实验

1. 读 `app_pair_bind.c` 函数列表，画配对时序图。
2. 搜索 `app_temp_password` 在开锁路径的调用。
3. 浏览器打开 JoyfulZone App，对照配对/临时密码界面。

---

## 易错点 / 困难点

- 配对码过期或只在一端 bind 成功。
- 临时密码只写 Host 未 RS485 同步 → 侧门不能用。
- 录入中途断电：Flash 与 AS608 可能不一致，需重录。
- 四端字段名不一致 → App 显示异常但门锁已开锁。

---

## 本课总结

- 配对连起门锁与 App 账号体系
- 临时密码是跨 Host/Client/后端的一致状态
- 产品闭环 = 硬件 + RS485 + 云 + UI 四段咬合

## 参考答案

1. `app_pair_bind.c`、`UI3_SCR_PAIR`、`JoyfulZone` App 与后端。
2. `TEMP_PWD_REVOKE`（0x07）。
3. `UI3_SCR_NFC_MGMT`（或旧 screen8 录入流）。
4. 手机 App 原型 + Node MQTT 桥后端，不是 MCU 固件。
