## 核心数据结构

```c
typedef struct {
    char acc[13];           /* 账号，含 '\0' */
    char pwd[11];           /* 密码 */
    uint8_t is_admin;       /* 1=管理员 */
    uint8_t active;         /* 1=启用 */
    uint8_t has_nfc;
    uint8_t nfc_uid[4];
    uint8_t has_fp;
    uint16_t fp_page_id_1;
    uint16_t fp_page_id_2;
} user_cred_t;
```

还有管理用 `user_auth_admin_t`，用于记录默认管理员是否被删等元数据。

---

## 功能怎么实现

| 操作 | 典型函数 |
|------|----------|
| 密码开锁 | `user_auth_unlock_credentials_match` |
| 管理员登录 | `user_auth_admin_credentials_match` |
| 按账号找索引 | `user_auth_find_index_by_acc` |
| NFC 查用户 | `user_auth_find_index_by_nfc_uid` |
| 指纹 page 查账号 | `user_auth_get_acc_by_fp_page` |
| 绑定 NFC | `user_auth_bind_nfc_by_acc` |
| 绑定指纹 | `user_auth_bind_fp_by_acc` |

### 存储与同步

- 内存中 `users[]` + `user_count`
- 修改后标记 dirty → **StorageTask** 写 W25Q
- RS485 `USER_ADD` / `USER_DELETE` 把变更推到从机

### 默认管理员兜底

- 出厂逻辑：账号 `1`，密码 `1111`
- Flash 读失败或无可登录管理员时，RAM 补写，**不清空已有数据**
- `default_admin_deleted` 等标志处理「删了 1 号但指纹还在模组」的边界

## 动手实验

1. 在 `app_user_ops.c` 搜索 `users_storage_save`，跟一次写 Flash 路径。
2. 用自检 `[AUTH]` 看 Flash 里 `1/1111` 是否 PASS。
3. 在代码里搜 `user_auth_unlock_credentials_match` 的调用点。

---

## 易错点 / 困难点

- `acc[13]`、`pwd[11]` 含结尾 `\0`，RS485 传二进制布局必须一致。
- 改用户后若立刻断电，可能丢最后一次异步写入（取决于 dirty 时机）。
- 前 32 用户槽与镜像策略相关，增删用户要想从机是否已同步。
- `user_auth_match_fp_page` 传一堆 `default_admin_*` 参数是为 deleted 管理员指纹兜底。

---

## 本课总结

- `user_cred_t` 是全项目用户真相来源
- `user_auth_*` 主从共用，保证鉴权一致
- 写 Flash 异步，读在启动时同步 load

---

## 本课提问

1. 管理员和普通用户在结构体里怎么区分？
2. 改密码后何时真正写入 W25Q？
3. `user_auth_get_acc_by_fp_page` 为什么需要 default_admin 参数？
4. Client 和 Host 共用 `user_auth_portable.h` 的好处？

---

## 参考答案

1. `is_admin` 字段为 1 表示管理员。
2. 标记 dirty 后由 Storage 任务异步刷盘。
3. 默认账号可能被删但指纹仍在 AS608，需把 page 映射回默认账号。
4. 主从鉴权规则一致，减少两边行为不一致的 bug。
