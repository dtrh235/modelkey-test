## 三条同步线

### 1. 用户增删

```
Host 修改 users[]
    → Storage 写 Flash
    → RS485 USER_ADD / USER_DELETE
    → Client 更新本地 users[]
```

### 2. 指纹模板镜像

```
Host 录入 StoreChar 成功
    → 读 512B 模板
    → FP_TEMPLATE × 4 片（128B/片）
    → Client 写入 AS608 同一 page_id
```

结构体 `rs485_fp_tpl_chunk_t`：

- `page_id`
- `chunk_idx`（0～3）
- `data[128]`

片间等待 `APP_FP_MIRROR_PAGE_GAP_MS`。

### 3. 全量镜像

```
Client 上电或发现不同步
    → MIRROR_SYNC_REQ
    → Host app_rs485_master_mirror_if_requested()
    → 排队下发全部用户 + 指纹模板
```

---

## 代匹配流程（复习+深化）

```
Client 侧门指纹开锁
    → GenImg / Img2Tz
    → FP_MATCH_CHAR 分片 → Host
    → Host Search + user_auth 查表
    → 应答匹配结果 + 账号
    → Client 开锁 + SLAVE_UNLOCK_NOTIFY
```

超时：`APP_FP_SLAVE_MATCH_RS485_MS`（默认 2000ms）。  
可选 fallback：`APP_FP_SLAVE_MATCH_FALLBACK`。

---

## 从机开锁通知

`rs485_unlock_notify_t`（15 字节）：

- `method_id`：1 密码 / 2 NFC / 3 指纹 / 5 临时密码
- `device_id`：从机固定 2
- `acc[12]`：账号

Host 收到后走上节云端上报链路。

## 动手实验

1. 双机烧录 Host+Client，Host 添加用户，示波器或日志看 RS485 活动。
2. 读 `app_fp_mirror_tx.c` 看重试与 gap 宏。
3. 搜索 `app_rs485_master_take_fp_commit` 理解镜像完成判定。

---

## 易错点 / 困难点

- 镜像与录入/MQTT 抢 RS485：看 `app_rs485_master_cmd_in_progress`。
- AS608 写模板慢，**不能连续无间隔发 4 片**。
- 从机 FpTask 须先写完队列再开锁（第 7 课）。
- 结构体布局主从必须一致，改字段要双端同改。

---

## 本课总结

- 主机是数据权威，从机是执行端
- 512B 模板必分片 + 间隔
- 代匹配保证 page→账号映射单一来源

## 本课提问

2. `rs485_fp_tpl_chunk_t` 里有哪些关键字段？
3. 主机如何知道指纹镜像写完？
4. 从机临时密码怎么同步？

---

## 参考答案

1. 上电或发现与主机不同步时（slave server poll 逻辑触发）。
2. `page_id`、`chunk_idx`、`data[128]`。
3. `app_rs485_master_take_fp_commit` 等提交状态接口。
4. `TEMP_PWD_SET` / `TEMP_PWD_REVOKE`（0x06 / 0x07）。
