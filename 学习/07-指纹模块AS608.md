## 专有名词

| 名词 | 解释 |
|------|------|
| **AS608** | 串口指纹模组，USART3 @ 57600 |
| **模板 Template** | 约 512 字节指纹特征 |
| **page_id** | 模组 Flash 指纹库槽位号 |
| **GenImg** | 采集指纹图像 |
| **Img2Tz** | 图像转特征 |
| **RegModel** | 合并两次特征为模板 |
| **StoreChar** | 模板写入模组指定 page |
| **Search** | 当前特征与库比对，返回 page_id |
| **代匹配** | 从机只采图传特征，主机 Search + 查用户表 |

---

## 功能怎么实现

### 主机录入

```
GenImg → Img2Tz(缓冲1) → 再采一次 → Img2Tz(缓冲2)
    → RegModel → StoreChar → 得到 page_id
    → user_auth_bind_fp_by_acc() 写入用户
    → RS485 FP_TEMPLATE 分片镜像到从机
```

### 主机指纹开锁

HomeAuth 轮询 → GenImg → Search → `user_auth_get_acc_by_fp_page()` → 开锁

### 从机代匹配（默认开启）

`APP_FP_SLAVE_MATCH_VIA_HOST=1`：

```
从机 GenImg/Img2Tz
    → RS485 FP_MATCH_CHAR 分片送主机
    → 主机 Search + 用户表判定
    → 应答结果 → 从机决定是否开锁
```

**为什么？** 用户表只在主机权威一份，从机不单独维护 page→账号映射，避免不一致。

### RS485 模板镜像

- 512B 分 4 片，每片 128B
- 结构体 `rs485_fp_tpl_chunk_t`：`page_id` + `chunk_idx` + `data[128]`
- 片间间隔 `APP_FP_MIRROR_PAGE_GAP_MS`，等 AS608 写完

## 动手实验

1. 在录入页录一枚指纹，串口（若开 `APP_FP_UNLOCK_LOG`）看 page_id。
2. 读 `app_rs485_fp_sync.h` 中分片结构体定义。
3. 对比 `APP_FP_SLAVE_MATCH_VIA_HOST` 为 0/1 时 Client 行为差异（阅读 `Client/main.c` FpTask）。

---

## 易错点 / 困难点

- 从机 `FpTask` **必须先处理** RS485 模板队列，再轮询开锁。
- 指纹模组 STA 脚 PE10 可读忙状态，录入中要等待。
- 同一用户可绑 `fp_page_id_1` 和 `fp_page_id_2`（两个手指）。
- 代匹配超时 `APP_FP_SLAVE_MATCH_RS485_MS`（默认 2000ms）。

---

## 本课总结

- AS608 走 USART3，录入得到 page_id 绑用户
- 从机默认「代匹配」，用户数据以主机为准
- 512B 模板经 RS485 4 片镜像到从机模组

---

## 本课提问

1. `APP_FP_SLAVE_MATCH_VIA_HOST=1` 时从机还本地 Search 吗？
2. 指纹模板多大？RS485 怎么传？
3. `fp_page_id_1` 和 `fp_page_id_2` 用途？
4. 从机 FpTask 为什么要先处理镜像队列？

---

## 参考答案

1. 不（或仅 fallback），特征送主机判定。
2. 512 字节；`FP_TEMPLATE` 命令，每片 128B，共 4 片。
3. 一个用户可绑两个指纹槽（如左右食指）。
4. 避免 RS485 下发的模板没写完就开锁，导致从机指纹库不完整。
