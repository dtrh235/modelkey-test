## 学习目标

- 掌握本项目专用「分层排障」模型
- 知道该开哪些调试宏、看哪个串口
- 建立「先硬件后软件、先数据后 UI」习惯

---

## 分层排障模型

```
L0  供电、下载线、BOOT 跳线
L1  自检工程 [W25Q] [TP] [NFC]
L2  数据层   [AUTH] 用户表 1/1111
L3  主程序   APP_BOOT_STAGE_LOG 卡在哪一步
L4  业务层   [RS485] [WiFi] [FP] 日志
L5  云端     Topic 流转、后端 .env、npm 是否运行
```

**原则**：在 Ln 未 PASS 前，不要跳到 Ln+2 猜 UI 逻辑。

---

## 串口用哪个？

| 场景 | 接口 | 波特率 |
|------|------|--------|
| 自检工程 | RS485（PC6/7） | 115200 |
| 主程序调试 | USART1 PA9/PA10 | 115200 |
| RS485 业务 | USART6 PC6/7 | 115200 |
| ESP8266 | USART2 PA2/3 | 115200 |
| 门锁执行 | UART4 PC10 | 115200 |

`APP_DEBUG_ON_USART6=1` 时调试口可能与 RS485 复用 USART6，以 `app_config.h` 为准。

---

## 常用调试宏

| 宏 | 作用 |
|----|------|
| `APP_BOOT_STAGE_LOG` | 启动阶段 `[BOOT]` |
| `APP_W25Q_DEBUG` | Flash 读写 |
| `APP_TOUCH_UART_DEBUG` | `[TP]` 触屏 |
| `APP_HOST_RS485_DIAG` | `[RS485]` |
| `APP_FP_UNLOCK_LOG` | 指纹开锁 |
| `APP_WIFI_CWJAP_TRACE` | WiFi 连接步骤 |

**注意**：`printf` 链入会增大 Flash，量产默认全 0。

## 典型案例速查

| 现象 | 第一层查什么 |
|------|--------------|
| 白屏 | BOOT 日志、`lv_refr_now`、背光 |
| 进不了开锁 | `[TP]` + 是否双击 |
| 1/1111 不对 | `[AUTH]` Flash 密码 |
| RS485 无响应 | 角色宏、接线、DE、共地 |
| App 无记录 | 从机是否 notify、MQTT、Topic 流转 |
| WiFi 连不上 | 三元组、CWJAP、信号 |

---

## 本课总结

- 先硬件后软件，先数据后 UI
- 日志标签是项目约定语言
- 不要一上来啃 `cloud_aliyun_at.c` 全文

---

## 本课提问

1. 主程序白屏第一步查什么？
2. RS485 不通先查哪四样？
3. 从机开锁 App 看不到记录查哪条链？
4. 触屏 PASS 但双击无效说明什么？

---

## 参考答案

1. `APP_BOOT_STAGE_LOG`、背光、`lv_refr_now`、是否卡在 `users_storage_load`。
2. A/B 线、GND 共地、波特率、DE 脚、主从地址与角色宏。
3. Client notify → Host RS485 收 → MQTT push → 后端 → App。
4. 硬件正常，问题在 UI 流程或用户不知道要双击。
